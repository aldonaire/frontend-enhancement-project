// About This Project

What is
CODE//RED?

A Git-first, design-forward student project that simulates how real software teams collaborate, build, review, and ship code.

The Mission

Learning by doing

This project bridges the gap between classroom theory and professional practice. Instead of building from scratch, students fork an existing codebase, work on branches, and submit pull requests — exactly like a real dev team.

The goal isn't just a finished webpage. It's to prove you can work within a collaborative Git workflow, write meaningful code, and document your process — skills that employers actively look for.

Learning Objectives

  • → Use Git branching effectively in a team context
  • → Apply frontend skills — HTML, CSS, JavaScript
  • → Improve UI/UX of an existing interface
  • → Perform pull requests and code reviews
  • → Resolve basic merge conflicts

Git Workflow

The journey visualized

Each stage builds on the last. Follow the timeline from fork to merged PR.

Step 01

Fork & Clone

Create your own copy of the instructor's repo and pull it to your local machine to start working.

Step 02

Create Feature Branch

Never touch main. Your branch is your workspace — named with your lastname for easy identification.

Step 03

Implement Enhancements

Three or more meaningful improvements across UI/UX, functionality, or optimization categories.

Step 04

Commit & Push

Write descriptive commits using the feat/fix/docs/style convention, then push to your fork.

Step 05

Open Pull Request

Target the instructor's main branch. Include a clear description of what you changed and why.

Step 06

Submit Documentation

Update the README with features added, tools used, and challenges encountered. This is graded.

Rules

The non-negotiables

✕

Do NOT push to main

All work must be on your feature branch. Pushing to main will break the project.

✕

No vague commit messages

"update file" is not acceptable. Describe the actual change made.

✕

Don't break existing functionality

Test everything before pushing. Your additions must not break what already works.

Deliverables

What to submit

01 — GitHub Branch

Your feature branch pushed to your forked repository with all commits visible.

02 — Pull Request

A clean PR targeting the instructor's main branch with a description of your changes.

03 — Documentation

README update or separate file covering features, tools, and challenges.
